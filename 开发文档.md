# DBTI 华语辩论人格测试开发文档

## 1. 项目目标

DBTI 是一个面向华语辩论圈的非严肃人格测试网页，参考 SBTI 的传播机制与娱乐化气质，做成“好笑、有梗、但又有一定贴合度”的辩论人格测试。

本项目不是心理测评，也不追求科学人格分类；核心目标是让用户完成若干道题后，得到一个和其辩论习惯、赛场风格、团队角色、表达偏好相关的戏谑型人格结果，并愿意截图或分享。

## 2. 参考对象与借鉴边界

参考站点：

- SBTI：https://cssbti.cn/
- LATI 景观缺德人格测试：https://af418.github.io/lati-landscape-test/
- 战锤40K人格测试：https://wh40k-test.xyz/

已研究到的参考机制：

- SBTI：静态页面结构，常规题 + 条件触发补充题，多维度计分，结果模板匹配，特殊人格/兜底人格，结果分享与人格图鉴。
- LATI：单页测试，题库和结果库内嵌，选项给类型加分，主人格 + 副人格，复制分享文案，风格化结果页。
- 战锤40K测试：独立题库 JSON 和角色 JSON，维度向量匹配，雷达图展示，截图分享，双语和更重的视觉氛围。

借鉴边界：

- 可以参考产品结构、计分思路、页面流程、分享机制。
- 不直接复制参考站的题目、结果文案、人格名称、视觉素材或代码实现。
- DBTI 的题库、人格标签、结果描述、分享文案和视觉风格均应原创。

## 3. 已确认决策

- 技术形态：静态单页网页，不做后端服务。
- 部署方向：可部署到 GitHub Pages、Vercel、Netlify 等静态托管平台，不需要额外购买服务器。
- 内容来源：题库、结果、人格标签和文案由本项目原创，但参考 SBTI 的荒诞、戏谑、传播感。
- 题量目标：正式版为 30 道常规题 + 1 道隐藏触发题，接近 SBTI 的 31 题体感。
- 开发节奏：先用 1 道题跑通最小流程，再扩充到完整题库。
- 计分方式：采用维度画像匹配，而不是简单类型加分。
- 内部匹配：倾向使用 16 个内部细维度，提高结果贴合度。
- 结果展示：使用 8 个展示维度绘制雷达图，可由 16 个内部维度汇总得到。
- 结果数量：当前 Demo 采用 22 个常规人格 + 1 个兜底人格；隐藏人格暂不实现，正式版再确认。
- 文案尺度：后续尝试温和版、正常版、缺德版三个版本，用实际体验决定最终风格。

## 4. 第一版功能范围

第一阶段先做最小闭环，不追求完整题库：

- 首页：展示项目名称、说明、测试入口。
- 答题页：展示当前题目、选项、进度、下一步按钮。
- 结果页：展示人格名称、人格代码、结果描述、维度雷达图、分享文案。
- 操作：重新测试、复制分享文案。
- 数据结构：预留 30+1 题、16 维匹配、8 维展示、多版本文案字段。
- Demo 阶段不实现隐藏人格，只保留正式阶段的扩展位置。

第一阶段暂不优先实现：

- 结果海报截图生成。
- 完整人格图鉴。
- 题目随机洗牌。
- 返回上一题。
- 多语言。
- 复杂动画系统。

上述功能可在最小闭环稳定后逐步加入。

## 5. 页面结构规划

静态单页建议拆成以下逻辑区域：

- `StartScreen`：首页/介绍页。
- `QuizScreen`：答题页。
- `ResultScreen`：结果页。
- `ProgressBar`：进度条。
- `RadarChart`：8 维雷达图。
- `ShareActions`：复制分享文案、后续可扩展截图分享。

因为使用原生静态网页，不一定需要真正的组件文件；但代码应按这些逻辑模块组织函数，避免所有逻辑混在一起。

## 6. 建议文件结构

第一版可使用如下文件结构：

```text
index.html
styles.css
app.js
data.js
需求文档.md
开发文档.md
```

各文件职责：

- `index.html`：页面 DOM 结构和基础入口。
- `styles.css`：视觉样式、响应式布局、动画。
- `app.js`：页面状态、答题交互、计分、结果渲染、复制分享。
- `data.js`：题库、人格库、维度定义、文案版本。

如果后续内容变大，可以把 `data.js` 拆成：

```text
data/
  questions.js
  types.js
  dimensions.js
```

## 7. 数据模型草案

### 7.1 维度源与实现原则

当前维度以 `测试维度.md` 为主文档。本次实现采用其中的 8 个核心展示维度和 16 个底层匹配维度，但做了两处工程化调整，需要在后续调参时持续确认：

- 调整 1：现有 30 道题暂不逐题改写为新字段，而是通过 `effectTransforms` 把旧题库语义字段转换成最终 16 个底层维度。这样可以保留已经跑通的题库结构，先验证新维度和 22 人格的整体效果。
- 调整 2：除“竞技导向”外，展示维度先按两个底层维度简单平均；“竞技导向”按短板加权计算，因为 `测试维度.md` 明确要求“表达—竞技倾向”和“赛场—队伍倾向”任一偏低时整体都应偏低。

### 7.2 内部维度

内部匹配使用 16 个底层维度，按 8 组组织。每组对应结果页雷达图的 1 个展示维度，但人格匹配仍使用 16 维向量，以提高结果贴合度。

| 展示维度 | 底层维度 A | 底层维度 B | 高分方向 |
| --- | --- | --- | --- |
| 竞技导向 | `expression_vs_competition` 表达—竞技倾向 | `stage_vs_team` 赛场—队伍倾向 | 更比赛脑、赢面脑、赛场脑。 |
| 压迫倾向 | `initiation_tendency` 开战倾向 | `pressing_intensity` 逼问强度 | 更主动开战、连续追问、压缩对手空间。 |
| 现实锚点 | `fact_vs_mechanism` 事实—机理倾向 | `reality_vs_setting` 现实—设定口味 | 更依赖事实材料、现实案例、政策与社会议题。 |
| 评委导向 | `self_vs_judge` 自我—裁判取向 | `judge_vs_performance` 裁判—表现归因 | 更围绕评委、裁判标准和票路组织比赛。 |
| 辩棍浓度 | `tournament_activity` 赛事活跃 | `daily_argumentativeness` 日常论辩 | 辩论参与频率和日常渗透度更高。 |
| 放飞程度 | `meme_intensity` 整活浓度 | `emotional_heat` 上头程度 | 更容易整活、外放、情绪升温。 |
| 协同倾向 | `team_construction` 整队建构 | `solo_vs_coordination` 单兵—配合倾向 | 更从整队分工、接线和同步视角理解比赛。 |
| 渲染倾向 | `chain_vs_scene` 链条—场景倾向 | `plain_vs_stylized` 朴素—包装倾向 | 更依赖场景、感染、包装和语言张力说服。 |

### 7.3 展示维度与合成公式

结果页雷达图展示 8 个核心维度，当前合成公式如下。公式中的字段值均为经过作答累加与相对中心化后的 0-100 分。

| 展示字段 | 展示名 | 合成公式 |
| --- | --- | --- |
| `competitive_orientation` | 竞技导向 | `round(0.7 * min(expression_vs_competition, stage_vs_team) + 0.3 * ((expression_vs_competition + stage_vs_team) / 2))` |
| `pressure_orientation` | 压迫倾向 | `round((initiation_tendency + pressing_intensity) / 2)` |
| `reality_anchor` | 现实锚点 | `round((fact_vs_mechanism + reality_vs_setting) / 2)` |
| `judge_orientation` | 评委导向 | `round((self_vs_judge + judge_vs_performance) / 2)` |
| `debate_immersion` | 辩棍浓度 | `round((tournament_activity + daily_argumentativeness) / 2)` |
| `abstraction_orientation` | 放飞程度 | `round((meme_intensity + emotional_heat) / 2)` |
| `collaboration_orientation` | 协同倾向 | `round((team_construction + solo_vs_coordination) / 2)` |
| `rendering_orientation` | 渲染倾向 | `round((chain_vs_scene + plain_vs_stylized) / 2)` |

### 7.4 旧题库 `effects` 转换层

当前题库选项仍使用上一版 16 个语义字段，例如 `attackPressure`、`frameworkModeling`、`memeDensity`。运行时通过 `effectTransforms` 转换为最终底层维度。总公式为：

```text
expandedEffect[targetDimension] = round(sum(sourceEffect[sourceKey] * effectTransforms[sourceKey][targetDimension]))
```

如果某个 `effects` 字段已经是最终 16 维字段，则按 `1.0` 原样进入该维度；不属于最终维度的字段会被过滤掉。

| 旧字段 | 当前转换公式 |
| --- | --- |
| `attackPressure` | `initiation_tendency += value * 0.7`; `pressing_intensity += value * 0.5` |
| `defensiveDissection` | `initiation_tendency += value * -0.35`; `pressing_intensity += value * -0.65`; `chain_vs_scene += value * -0.35` |
| `frameworkModeling` | `team_construction += value * 0.45`; `self_vs_judge += value * 0.25`; `chain_vs_scene += value * -0.25` |
| `criterionSensitivity` | `self_vs_judge += value * 0.75`; `judge_vs_performance += value * 0.5` |
| `evidenceDependence` | `fact_vs_mechanism += value * 0.75`; `reality_vs_setting += value * 0.5` |
| `logicPurity` | `chain_vs_scene += value * -0.75`; `fact_vs_mechanism += value * -0.3` |
| `expressionImpact` | `chain_vs_scene += value * 0.65`; `plain_vs_stylized += value * 0.35` |
| `punchlineImpulse` | `plain_vs_stylized += value * 0.75`; `meme_intensity += value * 0.35` |
| `impromptuCounter` | `initiation_tendency += value * 0.45`; `pressing_intensity += value * 0.25` |

### 7.5 题目结构

题目建议采用如下结构：

```js
{
  id: "q001",
  text: "自由辩刚开始，对方一辩抛出一个你觉得明显偷换概念的定义，你会：",
  options: [
    {
      label: "A",
      text: "立刻站起来追问，先把定义钉死再说。",
      effects: {
        initiation_tendency: 2,
        pressing_intensity: 2,
        self_vs_judge: 1
      }
    }
  ]
}
```

说明：

- `id`：题目编号。
- `text`：题干。
- `options`：选项列表。
- `effects`：该选项对内部维度的影响。后续新增题目建议直接使用最终 16 维字段；当前已写好的 30 题仍可使用旧字段，并由 `effectTransforms` 转换。

## 8. 计分与匹配方案

### 8.1 当前 Demo 计分流程

当前 Demo 已使用 30 道常规题、16 个底层维度、22 个常规人格。不实现隐藏人格，暂不启用兜底阈值。

计分流程：

- 初始化每个底层维度为 `50` 分。
- 读取用户选择的每个选项，将选项 `effects` 通过 `effectTransforms` 展开为最终 16 维分值。
- 对每个目标维度累加分值，单次累加后立即限制在 `0-100`：`rawScore[d] = clamp(rawScore[d] + expandedEffect[d], 0, 100)`。
- 记录每个维度是否被题目影响，得到 `touched[d]`，用于匹配时降低未触达维度的权重。
- 对 16 维做相对中心化，突出用户的相对倾向。
- 将归一化后的 16 维向量与 22 个常规人格模板做距离匹配。
- 将最终 16 维分数按 7.3 的公式合成 8 维雷达图。

### 8.2 相对中心化公式

当前归一化不是把所有分数线性拉满，而是围绕用户自己的 16 维均值做对比，避免“所有维度都偏高”时匹配到过于全面但不鲜明的人格。

```text
mean = sum(rawScore[d] for d in dimensions) / 16
score[d] = clamp(50 + (rawScore[d] - mean) * 1.45, 0, 100)
```

其中 `1.45` 是当前 Demo 的放大系数，用于让 30 题产生更明显的结果差异。后续如果体验上“测得太极端”，可下调到 `1.2-1.35`；如果结果区分度不足，可上调到 `1.5-1.7`。

### 8.3 人格匹配公式

每个人格都有一个 16 维理想画像 `profile[type][d]`。用户分数与人格画像距离越近，匹配度越高。

```text
weight[d] = touched[d] > 0 ? 1 : 0.15
distance[type] = sum(abs(score[d] - profile[type][d]) * weight[d] for d in dimensions)
weightTotal = sum(100 * weight[d] for d in dimensions)
similarity[type] = max(0, round((1 - distance[type] / weightTotal) * 100))
```

排序规则：

- 首先选择 `distance` 最小的人格作为主结果。
- 如果 `distance` 相同，选择 `similarity` 更高的人格。
- 第二名作为“相似人格”展示。
- 兜底人格数据已保留，但当前 Demo 暂不按阈值替换主结果；正式版可恢复“最高匹配度低于 60% 时进入兜底人格”的规则。

### 8.4 雷达图策略

结果页展示 8 维雷达图，使用原生 `canvas` 绘制，避免引入依赖。当前雷达图不直接参与人格匹配，只负责把 16 维结果压缩成更容易阅读的 8 维展示。

展示维度合成公式以 7.3 为准。除“竞技导向”采用短板加权外，其余维度先用简单平均，后续可根据试玩反馈再对某些维度做加权。

## 9. 内容策略

### 9.1 结果文案尺度

计划同时维护三种尺度：

- 温和版：更像友好侧写，适合广泛传播。
- 正常版：有梗、有锐评，但不明显冒犯。
- 缺德版：更接近 SBTI 的扎心和荒诞感，但仍避免真实人身攻击。

三版应尽量共享同一人格和同一计分逻辑，只替换表达方式，避免用户因为文案版本不同而得到不同结果。

三版文案的 UI 策略：

- 当前 Demo：已加入 `mild`、`normal`、`savage` 三版切换入口，方便内容调试阶段比较同一人格在温和版、正常版、缺德版下的表达效果。
- 正式上线阶段：可根据试玩反馈决定是否保留切换；如果追求更强的测试判定感，建议默认固定为“正常偏缺德”的主风格。

### 9.2 内容边界

允许使用：

- 辩位梗：一辩、二辩、三辩、四辩、自由辩、质询、结辩。
- 赛制梗：攻防、盘问、申论、评委判准、持方压力、赛前准备。
- 团队梗：资料组、约稿、教练、队友接线、社团训练、赛后复盘。
- 抽象表达：上头、查杀、救场、念稿、开框架、偷定义、刷数据。

应避免：

- 指名真实辩手、学校、赛事组织或具体争议。
- 基于地域、性别、身份、身体、疾病等方向的攻击。
- 直接复制 SBTI 或其他测试站的题目和结果文本。
- 使用明显低俗到影响公开传播的表达。

## 10. 视觉方向

第一版不照搬 SBTI 视觉，采用“赛场计时器 + 资料卡/判准表”的混合风格，做出华语辩论赛场氛围：

- 视觉关键词：赛场、计时器、判准表、资料卡片、辩题签、红蓝对抗。
- 首页：像一张赛前辩题/计时器界面，突出“DBTI 华语辩论人格测试”。
- 答题页：像一张评审表或资料卡，突出题目、选项和进度。
- 结果页：像一份“赛后人格判决书”，重点展示人格代号、锐评文案和雷达图。
- 页面气质：有一点正式赛事感，但结果页要有荒诞反差。
- 色彩方向：深墨黑/暗蓝黑底色，辩论红/辩论蓝主色，荧光黄或计时器红作强调色，纸张白/卡片米白作为内容区。
- 移动端优先：多数用户会在手机上完成测试和分享。
